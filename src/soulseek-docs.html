<h4 id="file-attribute-combinations">File Attribute Combinations</h4>

<p>These combinations are actively used by clients. Certain attributes can be missing if a file does not provide them.</p>

<ul>
<li>Soulseek NS, SoulseekQt (2015-2-21 and earlier), Nicotine+ (lossy formats), Museek+, SoulSeeX, slskd (lossy formats):
<ul>
<li>{0: <em>bitrate</em>, 1: <em>duration</em>, 2: <em>VBR</em>}</li>
</ul>
</li>
<li>SoulseekQt (2015-6-12 and later):
<ul>
<li>{0: <em>bitrate</em>, 1: <em>duration</em>} (MP3, OGG, WMA, M4A)</li>
<li>{1: <em>duration</em>, 4: <em>sample rate</em>, 5: <em>bit depth</em>} (FLAC, WAV, APE)</li>
<li>{0: <em>bitrate</em>, 1: <em>duration</em>, 4: <em>sample rate</em>, 5: <em>bit depth</em>} (WV)</li>
</ul>
</li>
<li>Nicotine+ (lossless formats), slskd (lossless formats):
<ul>
<li>{1: <em>duration</em>, 4: <em>sample rate</em>, 5: <em>bit depth</em>}</li>
</ul>
</li>
</ul>

but it handles the protocol well enough (and can be modified).</p>

<h3 id="modern-peer-connection-message-order">Modern Peer Connection Message Order</h3>

<p><em>Used by SoulseekQt, Nicotine+ 3.2.1 and later, Soulseek.NET-based clients (slskd, Seeker)</em></p>

<ol>
<li>User A sends <a href="#server-code-18">ConnectToPeer</a> to the Server with a unique token (indirect connection request)</li>
<li>User A sends a <a href="#peer-init-code-1">PeerInit</a> to User B (direct connection request)</li>
<li>The Server sends a <a href="#server-code-18">ConnectToPeer</a> response to User B with the same token.<br />
If User B receives the <em>PeerInit</em> message, a connection is established, and user A is free to send peer messages.<br />
Otherwise, once User B receives the <em>ConnectToPeer</em> message from the Server, User B proceeds with step 4.</li>
<li>User B sends a <a href="#peer-init-code-0">PierceFireWall</a> to User A with the token included in the <em>ConnectToPeer</em> message.<br />
If this succeeds, a connection is established, and User A is free to send peer messages.<br />
If this fails, User B retries for ~1 minute. If this still fails, no connection is possible, and User B proceeds with step 5.</li>
<li>User B sends a <a href="#server-code-1001">CantConnectToPeer</a> to the Server.</li>
<li>The Server sends a <a href="#server-code-1001">CantConnectToPeer</a> response to User A.</li>
</ol>

<p>Unlike SoulseekQt, Nicotine+ and Soulseek.NET-based clients skip step 5 in favor of letting the connection attempt time out on User A’s end.</p>

<h3 id="legacy-peer-connection-message-order">Legacy Peer Connection Message Order</h3>

<p><em>Used by Soulseek NS, Nicotine+ 3.2.0 and earlier, Museek+, soulseeX</em></p>

<ol>
<li>User A sends a <a href="#peer-init-code-1">PeerInit</a> to User B.<br />
If this succeeds, a connection is established, and User A is free to send peer messages.<br />
If this fails (socket cannot connect), User A proceeds with an indirect connection request (step 2).</li>
<li>User A sends <a href="#server-code-18">ConnectToPeer</a> to the Server with a unique token</li>
<li>The Server sends a <a href="#server-code-18">ConnectToPeer</a> response to User B with the same token</li>
<li>User B sends a <a href="#peer-init-code-0">PierceFireWall</a> to User A with the same token.<br />
If this succeeds, a connection is established, and User A is free to send peer messages.<br />
If this fails, User B retries for ~1 minute. If this still fails, no connection is possible, and User B proceeds with step 5.</li>
<li>User B sends a <a href="#server-code-1001">CantConnectToPeer</a> to the Server.</li>
<li>The Server sends a <a href="#server-code-1001">CantConnectToPeer</a> response to User A.</li>
</ol>

<h3 id="distribembeddedmessage">DistribEmbeddedMessage</h3>

<p>A branch root sends us an embedded distributed message. We unpack the distributed message and distribute it to our child peers.</p>

<p>The only type of distributed message sent at present is <a href="#distributed-code-3">DistribSearch</a> (distributed code 3).</p>
